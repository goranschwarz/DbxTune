GUI Tool

This is the simple option

Connect to a specififc DBMS instance using any of the DbxTune tools

  • SqlServerTune - To see what's happening right now in any Microsoft SQL Server instance.
  • AseTune - If you want to se how a Sybase ASE Instance is performing
  • PostgresTune - To look at how a Postres Instance is behaving
  • etc...

Move around in the various tabs or performance collectors to see counters for a specific section of the DBMS.

You can see the performance counters in various ways

  • Abs - Absolute counter values since the server last started or resetted the counters
  • Diff - Difference since previous sample/refresh
  • Rate - Difference per second since previous sample/refresh

For example:

  • Look at Active Statement to figgure out what SQL Statements that are currently running (and/or are blocking).
  • Look at Waits to figgure out the server or sessions are waiting for right now.
  • Look at Databases to check out space usage and other behaviour at a database level.
  • Look at Query Statistics find any statements that are costly or behavies strange.
  • Look at Missing/Unused Indexes sorted with some impact analyses.
The performance collectors varies depending on the database vendor

Draw conclutions from the above.
And fix the problem(s).

To see screenshots of the GUI Tool, please see the 'screenshot' section.

NO GUI Collectors

This is the way to go; If you want to have constantly monitoring (24*7) of all your database instances.

Decide what counters you want to save/store using the GUI mode for this will be good choise

Then all the counter details will be saved in a database. A new recording period will be started at midnight to simplify purging of old data

You can connect to any of the recorded sessions via the GUI Tool, if you want to view everything that you normally see with the GUI Tool

Or you can choose to install DbxCentral which is the Central Console and send performance trends to it...

Then you can look at trend graphs from a web browser

This requires a little bit more work and setup, but it's well worth the time...
Here you can see an overview how to setup DbxCentral

Alarm Handling

With the NO-GUI option you will also bennefit from Alarm Handling

When a performance collector pulls data from the monitored server, we can also check if specififc thresholds has been crossed, and send alarms if/when it happens

Below is a couple of examples

  • Running out of space in the database or the underlying file systems
  • We have long running transactions in any database
  • We have blocking locks in any database
  • Someone changed to configuration on the instance hopefully it was you
  • We are strting to run out of some resources in the instance
  • etc, etc...

Daily Summary Reports

At midnight: lets investigate what has happened in the recording period (last 24 hour)
Create a comprehencive report about todays activity.

Here are some examples what the Daily Summary Report usually contains

  • Information about what is recorded
  • Alarms: Active and the Alarm History for that day!
  • Overall statistics like: CPU Usage and What the server was WAITING on
  • Top SQL Statements, in various ways (CPU Usage, Waits, IO's etc)
  • Disk space usage
  • Top table activity
  • Database sizes
  • DBMS Configurations, and possible issues in configuration
  • etc, etc...

Daily Summary Reports

Daily Summary Reports are created at midnight, and will hold a summary of most important events for the last 24 hours

A Daily Summary Report will typically contain:

  • Usage charts for that last 24 hours
  • Wait statistics (what was the DBMS Server waiting on)
  • Top SQL Statements executed (typically on CPU usage, but also on other metrics)
    • Here table/index statistics are also included
    • Views used in the SQL Statement are resolved into table names
    • Execution plans (if collected) are also available for the SQL Statement
  • DBMS Specifics: Like for SQL Server Query Store information
  • Top Database usage (what database was using the most resources)
  • Operating System Counters... like: iostat, mpstat, etc (Windows OS Metrics are also available)
  • Top Database and Table sizes
  • Unused and Missing index list
  • DBMS Configuration (options that has been changed from default values)
  • etc, etc...

Daily reports are typically sent by mail (in a shorter version) when created...
Or they can be viewed from the DbxCentral web interface.

AseTune

Sybase/SAP Adaptive Server Enterprise

AseTune is the origin of all DbxTune, this is where it all Started.
At the time I was working at Sybase as a Performance trouble shooter, and I needed a tool to help me look at what was happening inside the DBMS Engine So I created AseMon, which was later on renamed to AseTune.

What is monitored

  • What SQL statements are currently beeing executed in Sybase ASE
  • How much CPU is Sybase ASE consuming for the moment
  • Database Space Usage, are we getting close to needing more space
  • Blocking locks, and any SQL Statements responsible for the block
  • What tables are the most accessed ones
  • What tables do I have locking conficts on
  • What is a specific spid/user doing for the moment
  • What is the Sybase ASE waiting for
  • And more...

Status of this tool
This is the most complete tool, and has been around for more than 15 years!

Download Info

The best way to get started with DbxTune is to simply download it start to use it!

The current download location is at SourceForge: https://sourceforge.net/projects/asetune/files/latest/download
Note: The project name at SourceForge is 'AseTune', but it contains the full DbxTune package with ALL available Collectors!

If you want latest development version, check Github or just contact me, and I'll give you an alternate download link.

Requirements

  • Java 8 and above

Installation Instructions

  • The download holds a ZIP file
  • Unpack the ZIP in some directory
  • Start any of the tools (files are located in the 'bin' directory)
    • Windows: start the BAT file (xxxTune.bat)
    • Linux/Unix/Mac: start the SH file (xxxTune.sh)
      Note: first have to change the shell scripts to be executable: chmod 755 bin/*.sh
  • Connect to any server you want to tune/monitor
If you want to setup NO-GUI Collectors and/or DbxCentral - The central Web Console to view Trend graphs, se the Github page for more info. Or contact me...
